# FinFact-BD: A Benchmark for Bengali Financial Misinformation Detection

> **Target venue:** FinNLP Workshop, COLING 2026

---

## Executive Summary

FinFact-BD is a benchmark for evaluating automatic detection of financial misinformation in Bengali. It pairs 10,000 original Bengali financial articles sampled from BENI v2 with claim-guided rewrites realized through controlled, multi-hop rewrite plans drawn from five core misinformation families. Rewrites are generated by Bangla seq2seq models (csebuetnlp/banglat5, Vacaspati/BanglaByT5) inside a planning and verification framework: a claim extractor selects editable financial propositions, a planner assigns the rewrite family and desired change, the generation model rewrites only the targeted sentence, and a multi-stage verifier checks claim integrity, surface fluency, and semantic quality before acceptance. The benchmark includes human validation (300 samples, three annotators), a held-out real-world test set, and evaluation across model families from classical baselines to closed LLMs. We formulate five research questions about what makes Bengali financial misinformation hard to detect, and release the full dataset and evaluation code for reproducibility.

This is not a dataset announcement. It is a controlled experimental setup designed to answer questions about model behavior on Bengali financial text.

Frozen release: `FinFact-BD-v1.0` regenerated on `2026-07-14`. Cite and evaluate this release version unless you explicitly need to compare against a future regenerated build.

## Research Questions

| ID | Question | Why It Matters |
|----|----------|----------------|
| **RQ1** | Which claim-rewriting families are hardest for current models? | Reveals specific linguistic weaknesses in Bengali NLP systems |
| **RQ2** | Do Bengali-specific PLMs outperform multilingual models on financial text? | Tests whether language-specific pretraining helps for domain-specific tasks |
| **RQ3** | How do open-weight LLMs compare to fine-tuned smaller models? | Probes the tradeoff between generalization and specialization |
| **RQ4** | Can models generalize from synthetic rewrites to real-world misinformation? | Validates ecological validity of the controlled rewriting approach |
| **RQ5** | What systematic errors do models make across rewrite families? | Enables targeted improvement in future work |

## Dataset

### Overview

FinFact-BD is constructed from BENI v2, a corpus of 38,107 Bengali economy articles. We select 10,000 articles, then generate controlled rewrites to create labeled real/fake pairs.

| Split | Original | Perturbed | Total |
|-------|----------|-----------|-------|
| Train | 8,000 | 8,000 | 16,000 |
| Validation | 1,000 | 1,000 | 2,000 |
| Test | 1,000 | 1,000 | 2,000 |
| **Total** | **10,000** | **10,000** | **20,000** |

### Rewrite Families

Five fact-aware strategies, each targeting a different financial proposition type. A single sample can compose 1, 2, or 3 of these operations into a multi-hop rewrite. Each strategy is realized through a Bangla generation model that rewrites the targeted sentence while preserving the surrounding article context.

| Type | Strategy | What It Tests |
|------|----------|---------------|
| Numerical Fact Change | Generation model rewrites a numeric claim in-place, producing a plausible but incorrect figure | Do models attend to explicit financial values? |
| Policy Reversal | Generation model flips the direction of a policy or market statement within the sentence | Can models detect reversals in financial actions? |
| Entity Replacement | Generation model swaps an entity with a same-class alternative in the rewritten sentence | Do models ground claims in the right institution or company? |
| Temporal Shift | Generation model changes a time reference to a different but plausible date | Do models track when the fact occurred? |
| Causal Inversion | Generation model reverses the cause-effect relationship in the targeted clause | Do models track cause-effect structure? |

### Difficulty Policy

Difficulty is family-aware rather than purely random. Policy reversal and entity replacement are biased toward easier one-hop edits, numerical and temporal changes center on medium difficulty, and causal inversion is biased toward harder multi-hop composition. The sampler also tracks global difficulty budgets so the final benchmark stays close to the intended 25/45/30 easy/medium/hard mix without introducing new rewrite operators.

Candidate selection is salience-aware as well: the generator prefers lead sentences and explicit claim-bearing spans such as numbers, dates, attribution phrases, policy verbs, and causal connectors. That makes the rewritten sample look more like a human-readable fake financial claim instead of a buried token-level edit.

The validation gate uses difficulty-aware contradiction thresholds so easy samples are not over-filtered by the same cutoff used for harder samples.
Because the rewrites are proposition-level edits, the semantic-similarity upper bound is intentionally permissive: we want to keep near-original factual rewrites and reject only pathological duplicates, not high-similarity misinformation.

### Quality Filtering

Quality control is built into the generation pipeline itself, not bolted on afterward.

**In-pipeline verification (primary):** Every generated rewrite passes through a three-stage verifier before acceptance:

1. **Claim integrity**: Did the intended claim change? Did any extra claim change? Can the changed span be localized to the planned sentence?
2. **Surface quality**: Is the rewrite fluent Bangla? Does it preserve journalistic style? Is it visually distinct from the source?
3. **Semantic quality**: Does the rewrite remain a believable Bangla financial news item? Does the contradiction score support the intended label? Are there duplicate artifacts?

If verification fails, the pipeline regenerates up to 3 attempts per sample. Only the best passing candidate is kept. Samples that fail all attempts are dropped with logged failure reasons.

**Downstream NLI filtering (optional):** mDeBERTa-v3 or another NLI model can be applied as an additional quality gate after generation, but it is no longer the sole quality control step. The in-pipeline verifier is the primary filter; mDeBERTa provides an independent cross-check for downstream consumers who want extra assurance.

**Validation:** The full pipeline is validated against human judgment (300 samples, 3 annotators). We report Fleiss' κ and model-human agreement to demonstrate reliability.

## Human Validation

### Protocol

- **Annotators:** 3 Bengali-native speakers with finance domain familiarity
- **Sample:** 300 articles (150 original, 150 perturbed), stratified across rewrite families
- **Task:** Claim-centric origin judgment (`original` / `perturbed` / `not sure`) with free-text justification and required confidence
- **Agreement:** Fleiss' kappa reported in paper

Annotators will work in separate XLSX workbooks. The `Samples` sheet shows the headline, a focused claim sentence, adaptive nearby context, and the full article text; the optional `Full Articles` sheet is still available for quicker scrolling when needed. Dropdown labels, required confidence, and free-text justification keep the annotation structured. A hidden `claim_type` column is reserved for later analysis only. A separate internal manifest keeps the sample-to-source mapping for later merging and analysis.

The workbook pack distributed in `data/finfact_bd/human_validation/` corresponds to the same frozen `FinFact-BD-v1.0` release.

### Purpose

The human annotation serves two roles:

1. **Gold standard** for evaluating model performance against human judgment
2. **Calibration check** for the mDeBERTa quality filter (comparing machine-filtered labels to human labels)

## Benchmark Models

We evaluate across five model families, ordered by increasing scale and complexity.

### Classical Baselines

| Model | Description | Training |
|-------|-------------|----------|
| TF-IDF + Logistic Regression | Sparse bag-of-words features, linear classifier | Supervised on train split |
| FastText | Subword embeddings with n-gram features | Supervised on train split |
| SVM | TF-IDF features with linear kernel | Supervised on train split |

### Bengali PLMs

| Model | Params | Description |
|-------|--------|-------------|
| BanglaBERT (`csebuetnlp/banglabert`) | 110M | Bengali-specific BERT, pretrained on Bengali web text |
| Bengali Electra | 110M | Bengali Electra, discriminative pretraining |

### Multilingual PLMs

| Model | Params | Description |
|-------|--------|-------------|
| mBERT | 110M | Multilingual BERT, 104 languages including Bengali |
| XLM-RoBERTa-base | 270M | Cross-lingual RoBERTa, 100 languages |
| ModernBERT | 395M | Updated encoder architecture with longer context |

### Open-Weight LLMs

| Model | Params | Fine-tuning |
|-------|--------|-------------|
| Llama-3-8B | 8B | LoRA (rank 16, alpha 32) |
| Qwen-2.5-7B | 7B | LoRA (rank 16, alpha 32) |
| Gemma-2-9B | 9B | LoRA (rank 16, alpha 32) |
| Mistral-7B | 7B | LoRA (rank 16, alpha 32) |

### Closed LLMs (Zero-Shot Only)

| Model | Access | Setting |
|-------|--------|---------|
| GPT-4o | API | Zero-shot classification |
| Gemini-1.5-Pro | API | Zero-shot classification |
| Claude-3.5-Sonnet | API | Zero-shot classification |

Closed models are evaluated zero-shot only. No fine-tuning, no few-shot prompting beyond the task instruction.

## Evaluation Plan

### Metrics

| Metric | What It Measures |
|--------|-----------------|
| Accuracy | Overall correct classification rate |
| Precision / Recall / F1 | Per-class performance (especially recall for misinformation) |
| Macro F1 | Balanced across classes regardless of size |
| Confusion Matrix | Error distribution across rewrite families |

### Evaluation Protocol

1. **Train/val/test split:** 80/10/10, grouped by `original_id` so no source article leaks across partitions
2. **Cross-validation:** 5-fold on training set for model selection
3. **Test set:** Held-out, reported once per model
4. **Real-world test set:** Separate set of actual Bengali financial misinformation (not synthetically perturbed) for ecological validity
5. **Statistical significance:** McNemar's test for pairwise model comparisons

### Error Analysis

Deep error analysis across four failure categories:

| Failure Type | Example | Question |
|--------------|---------|----------|
| **Numerical** | Model misses that revenue changed from ৫০০ to ১০০০ | Do models attend to digit tokens? |
| **Entity** | Model doesn't recognize a swapped company name | Are models grounded in real-world knowledge? |
| **Policy** | Model misses that a rate increase became a rate cut | Do models detect direction reversals? |
| **Temporal** | Model misses that the date moved from 2025 to 2024 | Do models track time references? |
| **Causal** | Model misses reversed cause-effect relationship | Do models parse discourse structure? |

Error analysis is conducted on the real-world test set to validate whether synthetic rewrite failures predict real-world failures.

### Kaggle Workflow

The claim-guided rewriting pipeline runs on Kaggle, where GPU access handles the generation models. The old rule-based perturbation pipeline (`src/generation/perturbation_pipeline.py`) is kept in the repo for reference but is no longer the primary generation path.

1. **Upload source data.** Add the BENI v2 source CSV to your Kaggle environment as a dataset.
2. **Open the rewriting notebook.** Open `scripts/finfact_bd_claim_rewrite_kaggle.ipynb`.
3. **Run all cells.** The notebook handles the full pipeline: claim extraction, claim selection, rewrite planning, Bangla generation (csebuetnlp/banglat5 or Vacaspati/BanglaByT5), multi-stage verification, and regeneration of failed samples.
4. **Download outputs.** The notebook exports `finfact_bd_rewritten_raw.csv`, `finfact_bd_rewritten_filtered.csv`, and `metadata.json`.

### Kaggle Quality Filtering

For downstream NLI filtering after generation:

1. Upload `data/finfact_bd/finfact_bd_rewritten_raw.csv` as a Kaggle Dataset
2. Open `scripts/finfact_bd_mdeberta_filter_kaggle.ipynb`
3. Run all cells. Download filtered output.

### What This Is Not

- Not free-form fake news generation
- Not an unconstrained paraphraser
- Not a token-replacement script
- Not a classifier
- Not article-level rewriting without a plan

The generation model is an executor inside a larger planning and verification pipeline. The planner decides what claim should change, the generator realizes the change in one targeted sentence, and the verifier decides whether the result is acceptable.
